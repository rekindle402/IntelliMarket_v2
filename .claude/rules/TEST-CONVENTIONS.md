# 테스트 작성 컨벤션

## 단위 테스트 vs 통합 테스트

| 구분 | 검증 대상 | 의존성 처리 | 도구 | 속도 |
|------|-----------|-------------|------|------|
| 단위 테스트 | 클래스 하나의 로직 | Mockito로 외부 의존성을 가짜(mock)로 대체 | JUnit5 + Mockito | 빠름 |
| 통합 테스트 | 여러 계층이 실제로 연결되어 동작하는 흐름 (Controller ~ DB) | 실제 빈/DB를 띄움 | `@SpringBootTest` + `MockMvc` | 느림 |

> 컨트롤러가 `AuthenticationManager` 등 프레임워크 컴포넌트에 위임하는 구조라면, 단위 테스트는 "직접 작성한 로직이 있는 클래스" 위주로 선정한다 (예: `CustomUserDetailsService`, 이벤트 리스너 등)

---

## 테스트 대상 선정 기준

> 빠른 완성을 위해 AI가 아래 기준에 따라 테스트 대상을 선정하고 의도를 설계한다.

1. **복잡도가 높은 API/로직 우선** — 단순 CRUD보다 여러 컴포넌트가 얽히거나 예외 분기가 많은 대상을 우선 선정
2. **보안/정합성과 직결된 로직** — 예: 로그인 시 이메일/비밀번호 오류를 구분하지 않고 동일한 메시지로 응답하는지
3. **상태 변화 및 부수효과가 있는 로직** — 단순 조회보다 생성/수정/삭제, 이벤트 기반 부가 작업(예: 로그인 성공 후 `lastLoginAt` 갱신) 등

---

## 진행 순서

테스트 피라미드 원칙에 따라 단위 테스트를 우선한다.

1. 단위 테스트 먼저 작성 — 직접 작성한 로직을 빠르고 독립적으로 검증, 실패 시 원인 파악이 명확함
2. 이후 통합 테스트로 계층 간 연동, 설정, 전체 흐름을 보완 검증

## 테스트 환경 구성

- 운영 환경(MySQL)과 동일한 DB로 통합 테스트를 진행하기 위해 별도 테스트 스키마(`im_v2_test`)를 사용한다
- `src/test/resources/application-test.properties`에서 `@ActiveProfiles("test")`로 분리된 설정을 사용한다
- DB 접속 정보(`DB_USERNAME`, `DB_PASSWORD`)는 평문으로 작성하지 않고 환경 변수(`${DB_USERNAME}`, `${DB_PASSWORD}`)로 주입한다
- Flyway 마이그레이션을 그대로 재사용하여 운영 환경과 스키마 정합성을 보장한다

## 네이밍 및 구조

| 항목 | 규칙 | 예시 |
|------|------|------|
| 클래스명 | `{대상클래스명}Test` | `AuthControllerTest` |
| 메서드명 | `{기능}_{상황}` (영어) | `login_success`, `login_fail_wrongPassword` |
| `@DisplayName` | 한글로 "어떤 상황에서 무엇을 검증하는지" 명시 | `"비밀번호가 일치하지 않으면 인증 실패 응답을 반환한다"` |
| 패키지 위치 | `src/main`의 도메인 패키지 구조를 그대로 미러링 | `domain/auth/controller` → `test/.../domain/auth/controller` |

## 작성 기법

- AAA 패턴(Arrange-Act-Assert)으로 구성: 준비 → 실행 → 검증의 흐름이 드러나도록 작성
- 통합 테스트는 `@Transactional`을 적용하여 각 테스트 후 DB 변경사항을 롤백, 테스트 간 데이터 독립성을 보장한다
- 공통 준비 작업(예: 회원가입 후 로그인)은 `@BeforeEach` 또는 `private` 헬퍼 메서드로 추출하여 중복을 줄인다
- 응답 형식은 `api-conventions.md`의 `ApiResponse<T>` 래퍼 구조를 기준으로 검증한다 (`$.success`, `$.data.*`, 에러 시 `$.code`, `$.message`)
